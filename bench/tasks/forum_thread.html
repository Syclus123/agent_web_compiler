Is Rust the right choice for web backends in 2025?

rust web backend 2,847 views 34 replies
rustacean42 Regular 142 posts
June 15, 2025, 10:30 AM

I have been building web backends in Python (FastAPI) and Node.js (Express) for the past 6 years. Our team is evaluating Rust with Axum for a new high-throughput microservice that needs to handle 50K requests per second with p99 latency under 10ms.

Our current Python service tops out at about 5K req/s on similar hardware, and the Go alternative we benchmarked gets to about 30K req/s. The Rust prototype using Axum hit 65K req/s with p99 at 3ms, which is impressive.

However, I am concerned about:

  • Developer productivity — the borrow checker is a steep learning curve for our team
  • Ecosystem maturity — are the ORM and database libraries production-ready?
  • Hiring — can we find Rust developers, or do we need to train?
  • Compile times — our CI pipeline already takes 20 minutes

Has anyone made this transition? What were the real-world trade-offs? I would love to hear from teams that have run Rust web services in production for more than a year.

We switched our main API from Node.js to Axum about 18 months ago. Here is our honest assessment:

Pros:

  • Performance is extraordinary. We went from 12 servers to 3 for the same traffic.
  • Memory usage dropped from 2GB to 180MB per instance.
  • The type system catches so many bugs at compile time that our production error rate dropped 80%.
  • sqlx is excellent and fully production-ready for PostgreSQL.

Cons:

  • Initial development velocity dropped about 40% for the first 3 months.
  • Compile times are real — we use cargo-watch and sccache to mitigate.
  • Some ecosystem gaps remain. Email libraries and OAuth integrations are less polished than Node equivalents.

Has anyone made this transition? What were the real-world trade-offs?

Overall, I would say it is absolutely worth it for performance-critical services. For typical CRUD APIs, Python or Go are still more pragmatic choices. Choose based on your actual bottleneck, not hype.

Re: ecosystem maturity — here is what I have found reliable in production:

  • sqlx — compile-time checked SQL queries, async, excellent for Postgres
  • sea-orm — full ORM if you prefer that pattern, very active development
  • tower — middleware framework, on par with Express middleware
  • serde — serialization is world-class, best-in-class JSON handling
  • tracing — structured logging/observability, better than most Node options

The weak spots are mostly in higher-level application frameworks. There is no Django equivalent where everything is batteries-included. You will assemble your stack from individual crates.
